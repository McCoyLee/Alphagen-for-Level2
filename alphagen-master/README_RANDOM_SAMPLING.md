# Random-Window Episodic Training (`claude/random-sampling-training-data-FLuf8`)

Drop-in alternative to `scripts/rl_tick_rolling.py` that swaps the walk-forward
schedule for **random-window episodic sampling** with **risk-adjusted reward**
and **periodic Genetic-Programming evolution** of the factor pool.

> Entry point: `scripts/rl_random_sampling.py`
> All new modules live in `alphagen_level2/`.  Nothing in the original tick
> training pipeline is modified.

---

## 1. What changed

| Concern | Before (rl_tick_rolling) | After (rl_random_sampling) |
|---|---|---|
| Training data | Walk-forward 6m/2m/2m windows | Random 1200-bar slices from the full 3-year dataset, fresh per episode |
| Reward signal | `╬▒┬╖tanh(\|IC\|) + ╬▓┬╖tanh(r╠Д) тИТ ╬│┬╖tanh(tc)` | Sortino on the future 100-bar bar-by-bar PnL, plus drawdown / fat-tail / kurtosis penalties, decorrelation bonus, complexity penalty |
| Factor pool | Capacity configurable, RL-only updates | Capacity 20 (default), LLM-seeded, RL-mined, and **GP-evolved** every N PPO rollouts |
| Cache | `./out/tick_cache` (HDF5 тЖТ bar tensor pickle) | `./out_random/tick_cache` (same mechanism, new default path) |
| Convergence plot | `convergence_logger.py` (IC-centric, train/valid/test) | `random_metrics.py` (reward / risk / IC, **pool best vs pool avg** overlays + GP activity) |

## 3. Episode contract

Each episode now corresponds to a single completed expression generated by
the RL policy.  When the env terminates, the candidate is scored as follows:

1. **Pick a window**.  `RandomWindowSampler` chooses a random start
   `t0 тИИ [bt + history_buffer, T тИТ future_margin тИТ window_bars)` and builds a
   `TickStockData` *view* of `[history_buffer | window_bars | forward_margin]`
   that shares the parent's tensor (no I/O, no GPU copy).
2. **Compute risk components** on the future-100-bar bar-by-bar PnL series
   (mid-price returns, position = `clip(zscore, ┬▒3)/3`, direction =
   `sign(IC)` measured on the same window).
3. **Shape the reward**:

   ```
   reward = w_sortino ┬╖ tanh(Sortino)
          + w_sharpe  ┬╖ tanh(Sharpe)
          + w_ic      ┬╖ tanh(rank_IC / 0.05)
          тИТ w_dd      ┬╖ tanh(max(0, MaxDD тИТ 0.02) / 0.02)
          тИТ w_fat     ┬╖ tanh(max(0, max|bar_ret| тИТ 0.01) / 0.01)
          тИТ w_kurt    ┬╖ tanh(max(0, ExcessKurt тИТ 8) / 8)
          + w_decorr  ┬╖ (1 тИТ avg|IC| with current pool)
          тИТ w_compl   ┬╖ expr_size
   ```
4. The reward is averaged over `score_windows` (default 4) random windows
   to dampen sampling noise; this is also the reward returned to PPO.

Backtest alignment is preserved: per-bar PnL uses the same
`p ┬╖ r тИТ cost ┬╖ |╬Фp|` decomposition as `backtest_tick_pnl.py` (with
`turnover_cost = 0.0006`), so the reward and the backtest agree on what
"good" looks like.

## 4. Factor-pool management (capacity = 20)

* **LLM init** тАФ exactly the existing flow.  We instantiate a
  `DefaultInteraction` against the existing `get_tick_system_prompt()` and
  call `inter.run(n_updates=4)`.  The pool factory is a lightweight
  `_refresh_pool` that resets the pool's slots and `force_load_exprs`.
* **Seed fallback** тАФ if the LLM produces fewer than 5 valid expressions
  (network blip, parser rejects), 10 hand-written seed expressions are
  loaded so RL has something to mine.
* **RL mining** тАФ `MaskablePPO` generates token-sequence candidates; each
  completed expression goes through `pool.try_new_expr`, which now calls
  `resample_window()` first so every candidate is judged on a fresh slice.
* **GP evolution (toggleable)** тАФ every `gp_every_n_epochs` PPO rollouts a
  `GPCallback` runs `evolve_pool`:
  * Tournament-select two parents from the pool by composite score.
  * Either subtree-crossover or random-subtree mutation.
  * Score the offspring via `pool.score_candidate` (same risk-adjusted
    reward path).
  * If the offspring beats the current worst pool member (and we haven't
    exceeded `replace_worst_n` evictions this generation), evict and add.
  * `gp_history.json` records every generation: epoch, accepted exprs,
    pool size before/after.
* **Decorrelation** тАФ each `_score_expr` call computes the average mutual
  IC of the candidate against current pool members on the same window;
  high pairwise correlation drains the `decorrelation_bonus` term.

## 6. Convergence plot

`random_metrics.py` records a row at the end of *every* PPO rollout into
`convergence.csv`.  At the end of training (and on demand) it renders a
2├Ч3 figure to `convergence.png`:

| Panel | Contents |
|---|---|
| (0,0) Composite Reward | `pool_best_reward` vs `pool_avg_reward` over the 20 pool members |
| (0,1) Risk Score | best vs avg of the recent risk-reward proxy (last `pool_size` evaluations) |
| (0,2) \|Rank IC\| | best vs avg over the same window |
| (1,0) Pool Growth | pool size + cumulative `eval_cnt` (twin-axis) |
| (1,1) Recent Episode Reward | mean ┬▒ 1╧Г of the last 256 episode rewards |
| (1,2) GP Activity | cumulative offspring accepted by the GP loop |

The two main asks тАФ `pool_best` and `pool_avg` тАФ are present on every
quality panel of the top row.  Best uses `np.nanmax` over the live
composite scores stored in `_composite_scores[:size]`; avg uses
`np.nanmean`.  The recent-reward panel is signal-of-the-policy rather
than the pool, useful for spotting regressions while training.

To re-render without retraining:

```python
from alphagen_level2.random_metrics import plot_random_sampling_curves
plot_random_sampling_curves("./out_random/convergence.csv")
```

```bash
python scripts/rl_random_sampling.py \
    --data_root=~/EquityLevel2/stock \
    --instruments='["510300.sh"]' \
    --start_time=2023-01-01 --end_time=2025-12-31 \
    --total_steps=300000 \
    --pool_capacity=20 \
    --score_windows=4 \
    --gp_enabled=True --gp_every_n_epochs=20 \
    --gp_offspring=10 --gp_replace_worst_n=5 \
    --window_bars=1200 --future_bars=100 --lookback_bars=200 \
    --sortino_weight=1.0 --max_dd_threshold=0.02 --fat_tail_threshold=0.01
```

| Flag | Default | Meaning |
|---|---|---|
| `--cache_dir` | `./out_random/tick_cache` | parsed-tensor cache; `None` to disable |
| `--save_root` | `./out_random` | all outputs land here |
| `--window_bars` | 1200 | observation window length (1h @ 3s) |
| `--future_bars` | 100 | reward horizon (~5 min @ 3s) |
| `--lookback_bars` | 200 | rolling z-score window; **must satisfy** `lookback + future + delay + 2 тЙд window_bars` |
| `--score_windows` | 4 | windows averaged per `_score_expr` (variance vs cost) |
| `--pool_capacity` | 20 | initial / steady-state factor pool size |
| `--gp_enabled` | True | toggle GP evolution |
| `--gp_every_n_epochs` | 20 | GP cadence in PPO rollouts (10тАУ50 reasonable) |
| `--llm_warmstart` | True | run LLM init via existing `DefaultInteraction` |

## 8. Behavioural notes & guardrails

* `RandomWindowSingleFactorPool` prints a constructor-time warning if
  `lookback + holding + delay + 4 > window_bars` тАФ this geometry would make
  every reward computation early-exit (the smoke-test bug we hit).
* `_calc_ics` is overridden to NaN-reject expressions whose forward-return
  data can't be prepared on the current window; this prevents silent
  insertion of degenerate factors with `composite=0 / direction=+1`.
* The first time `_score_expr` returns no-data it emits a one-shot
  `WARNING` so future smoke tests fail loudly rather than producing
  empty `reward_stats.json`.
